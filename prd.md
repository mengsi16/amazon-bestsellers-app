## 1. 产品概述

### 1.1 产品背景

在亚马逊跨境电商选品场景中，运营人员需要频繁分析特定细分类目的畅销商品（Best Sellers），了解竞争格局、功能趋势、价格带分布以及潜在差异化机会。传统人工分析耗时高、标准不统一，而纯抓取工具通常只能提供原始数据，难以直接形成结构化商业洞察。

`amazon-bestsellers-app` 最初源于一个基于 Claude Code CLI 的自动化分析原型，已经验证了从抓取、清洗、分块、维度分析到汇总报告的端到端链路。当前目标是将其产品化为一个可稳定运行的 Web 应用，供内部团队持续使用，并在低成本前提下实现可重复、可追踪、可维护的分析流程。

### 1.2 产品目标

- **自动化选品洞察**：输入一个亚马逊 Best Sellers 类目 URL，自动输出结构化的类目分析报告。
- **低成本增量刷新**：对已分析类目支持低成本刷新，仅处理新增商品与关键变化。
- **支持全量重新分析**：允许用户在已有历史数据存在时清空旧状态并从头重新跑完整分析。
- **Web 化交付**：用户通过浏览器创建任务、查看实时进度、阅读历史报告并下载结果。
- **工程化稳定性**：后端统一管理任务状态、workspace 路径、Claude session 与历史记录，避免路径漂移、状态污染和数据遗失。

### 1.3 产品定位

一个面向跨境电商团队的 **Amazon 细分类目选品分析 SaaS 工具的开源自托管版本**。核心用户为需要高频监控品类变化、评估进入机会、支持选品决策的选品经理和运营专家。

## 2. 用户与场景

### 2.1 目标用户画像

- **选品经理**：每日监控多个核心类目，关注新品入榜、竞争结构变化和进入机会。
- **运营专家**：非技术背景，希望直接看到结构化结论，而不是手动处理抓取数据。
- **系统管理员 / 开发者**：负责部署和维护，希望后端、Agent、MCP 工具和 workspace 规则清晰稳定。

### 2.2 典型使用场景

| 场景 | 描述 | 对应功能 |
| :--- | :--- | :--- |
| **首次分析** | 用户输入类目 URL，首次生成完整类目报告 | 全量分析 |
| **日常监控** | 用户希望获取新品、排名变化和最新结论 | 增量刷新 |
| **深度重跑** | 用户怀疑旧数据受脏状态影响，要求完全重跑 | 全量重新分析 |
| **报告回溯** | 用户重新打开历史任务查看旧报告 | 任务列表、报告浏览 |
| **失败恢复** | 任务中断后用户希望继续执行未完成部分 | 继续分析 |

## 4. 核心功能需求

### 4.1 全量分析（Full Analysis）

- **输入**：一个亚马逊 Best Sellers 类目 URL。
- **处理流程**：
  1. 后端解析 URL，提取 `browse_node_id`。
  2. 后端创建 `workspace/{browse_node_id}/` 并注册任务。
  3. Agent 爬取榜单，获取 Top 商品 URL / ASIN / 排名。
  4. 爬取商品详情页并沉淀原始数据。
  5. 执行分块、抽取、审计、维度分析和最终汇总。
- **输出**：
  - `reports/` 目录中的维度报告
  - `summary.md`
  - `.analysis_meta.json`
- **交付形式**：Web UI 在线预览并支持下载。

### 4.2 增量分析（Incremental Refresh）

- **前置条件**：该类目已完成过至少一次成功分析，并存在可复用的 workspace。
- **处理原则**：
  1. 读取历史元数据和榜单快照。
  2. 重新抓取最新榜单。
  3. 识别新增 ASIN、跌出 ASIN、排名变化。
  4. 仅对新增或必要变化的 ASIN 进行详情抓取和后续处理。
  5. 复用已有中间结果，重写受影响报告与 `summary.md`。
- **成本目标**：显著低于全量分析，目标为全量成本的 10% 以下。

### 4.3 全量重新分析（Full Re-analysis）

- **使用场景**：
  - 用户怀疑旧数据、旧 session 或中间产物已污染结果。
  - 用户明确要求“从头重新跑一遍”。
  - 已有分析存在，但希望得到完全独立的一轮新结果。
- **处理流程**：
  1. 后端清除 workspace 中会影响本轮结果的旧产物。
  2. 清空 `.analysis_meta.json` 中的陈旧 `session_id` 和运行状态。
  3. 删除与该任务相关的历史流式记录。
  4. 以全新 Claude session 重新启动完整分析流程。
- **前端要求**：
  - 提供独立于“刷新排名”的按钮。
  - 弹出强提醒确认框。
  - 明确提示“将覆盖旧结果且不可恢复”。

### 4.4 任务管理

- **创建任务**：用户提交 URL 后创建唯一任务 ID，并启动异步分析。
- **实时进度推送**：通过 SSE 推送阶段进度和流式输出。
- **任务列表**：展示状态、创建时间、更新时间、类目标识。
- **任务操作**：
  - 继续分析
  - 刷新排名
  - 全量重新分析
  - 删除任务

### 4.5 报告查看与下载

- 在线预览 Markdown 报告
- 查看维度报告和 `summary.md`
- 下载报告文件
- 后续可扩展为结构化中间数据导出、HTML / PDF 导出

## 5. 非功能需求

### 5.1 性能要求

- 全量分析目标耗时：10-30 分钟 / 单类目
- 增量刷新目标耗时：3-10 分钟 / 单类目
- SSE 推送延迟：2 秒内

### 5.2 可靠性

- 任务状态必须可持久化。
- 服务重启后任务信息不可丢失。
- 删除任务后不得保留会影响后续重跑的陈旧 `session_id`。
- 后端必须是 workspace 路径的唯一权威来源。
- Agent 长时间无输出时应支持后续接入自动超时终止机制。

### 5.3 成本控制

- 增量刷新必须尽可能复用已有数据，避免重复抓取与重复分析。
- 原始 HTML 应通过 MCP 和 chunker 降低传入 LLM 的体积。
- 后续支持按不同步骤切换不同成本级别模型。

### 5.4 可维护性

- 后端、Agent Prompt、MCP 工具分层清晰。
- Prompt 与业务逻辑尽量解耦。
- 关键元数据格式必须稳定演进。
- workspace 管理规范必须文档化并强制执行。

## 6. 系统架构设计

```
[用户浏览器]
     │  HTTP / SSE
     ▼
[FastAPI Web 后端]
     │
     ├── 任务状态持久化（当前：tasks.json + SQLite history）

     ├── workspace 路径管理（权威）
     └── SSE / 报告读取 / 删除 / 下载

     ├── orchestrator agent

     │
     ▼

  ├── categories/
  ├── products/
  ├── golden/
  ├── chunker/
  ├── chunks/
  ├── reports/
  ├── summary.md
  └── .analysis_meta.json
```

## 7. 后端 API 定义

> 说明：当前项目实际 API 前缀为 `/api`，本 PRD 优先记录当前实现，后续版本化再统一迁移到 `/api/v1`。

| 方法 | 路径 | 描述 |
| :--- | :--- | :--- |
| `POST` | `/api/tasks` | 创建全量分析任务 |
| `GET` | `/api/tasks` | 获取任务列表 |
| `GET` | `/api/tasks/{task_id}` | 获取任务详情 |
| `GET` | `/api/tasks/{task_id}/progress` | SSE 流式进度推送 |
| `GET` | `/api/tasks/{task_id}/reports` | 获取报告与 phase 状态 |
| `GET` | `/api/tasks/{task_id}/history` | 获取历史 stream / chat 记录 |
| `POST` | `/api/tasks/{task_id}/resume` | 继续分析 |
| `POST` | `/api/tasks/{task_id}/refresh` | 增量刷新 |
| `POST` | `/api/tasks/{task_id}/reanalyze` | 全量重新分析 |
| `DELETE` | `/api/tasks/{task_id}` | 删除任务与关联状态 |

### 7.1 API 基本要求

- 删除任务必须清理会影响后续重跑的陈旧 `session_id`。
- 任务详情必须返回 `workspace_path`，供前端统一展示和定位。
- SSE 事件需要支持阶段状态恢复和错误展示。
- `refresh` 与 `reanalyze` 必须是语义明确的两个不同操作。

## 8. Agent 模板设计

### 8.1 当前推荐方案

v1 继续采用 **单编排器 + 模式切换** 的设计：

- 普通任务：全量分析模式
- `refresh`：增量更新模式
- `reanalyze`：清空状态后重新进入全量分析模式

该方案更贴近当前代码，改造成本更低。

### 8.2 后续演进方案

后续可演进为两个独立 Prompt：

- `full-analysis-agent.md`
- `incremental-agent.md`

用于降低 Prompt 复杂度，提高可维护性和可测试性。

### 8.3 Agent 强制规则

- Agent 不得自行决定 workspace 根目录。
- 若上层明确提供 `workspace 绝对路径`，Agent 必须无条件服从。
- 禁止根据 CWD、session context、系统上下文推导其它目录。
- 刷新模式不得重新抓取未变化 ASIN。
- 全量重新分析不得复用旧 session 继续运行。

## 9. 工作空间路径管理规范

### 9.1 后端权威原则

- 后端根据 URL 提取 `browse_node_id`。
- 后端负责计算唯一权威 `workspace_path`。
- 后端启动 Agent 时必须显式传入：
  - `browse_node_id`
  - `workspace 绝对路径`
- 前端、报告读取、删除、下载均基于该路径。

### 9.2 风险背景

历史问题表明，如果 Agent 根据错误的 CWD 推导 workspace，可能出现：

- 任务记录路径与实际写入路径不一致
- 前端看不到报告与下载内容
- 删除任务后残留的 `session_id` 污染后续分析
- 数据写到错误项目目录或错误盘符

### 9.3 推荐增强措施

- 后端写入 `.workspace_root` 记录 `task_id` 与 `browse_node_id`
- 后端通过环境变量 `ANALYSIS_WORKSPACE` 传递路径
- Prompt 中重复声明 `workspace 绝对路径`，形成双保险

## 10. 元数据模型

`.analysis_meta.json` 建议包含以下字段：

```json
{
  "task_id": "141e0ec8",
  "browse_node_id": "1040658",
  "session_id": "20353c41-95bc-4a5e-9a82-a681eb1b720e",
  "analyzed_at": "2026-04-28T10:00:00Z",
  "asins": ["B0XXXXXXX", "B0YYYYYYY"],
  "ranking": {
    "1": "B0XXXXXXX",
    "2": "B0YYYYYYY"
  }
}
```

### 10.1 元数据使用要求

- 刷新时用于比对新旧 ASIN 与排名变化。
- 恢复任务时可用于识别是否存在可复用 session。
- 删除任务和全量重新分析时必须清空会污染后续运行的 `session_id`。

## 11. 任务状态机

```
[pending] --> [running] --> [completed]
     │            │
     │            └--> [failed]
     │
     └--> [failed]
```

### 11.1 状态要求

- 创建任务时写入 `pending`
- 子进程启动后进入 `running`
- 报告和 `summary.md` 落盘后才可标记 `completed`
- 子进程异常退出、超时或关键文件缺失时标记 `failed`
- `resume` 只用于可恢复任务
- `refresh` 只用于已有历史结果的任务
- `reanalyze` 在清空状态后重新进入 `pending`

## 12. 当前关键 Gap 分析

### 12.1 前端

#### 高优先级

| Gap | 说明 |
| :--- | :--- |
| **无任务搜索 / 过滤** | 历史任务增多后难以定位 |
| **无报告对比** | 无法横向比较两个类目报告 |
| **无完成通知** | 用户需要长时间盯浏览器 |
| **无 ETA / ASIN 级进度** | 当前只有阶段级状态，无法判断具体卡点 |
| **无成本展示** | 用户无法感知刷新与全量分析的成本差异 |

#### 中优先级

| Gap | 说明 |
| :--- | :--- |
| **无原始数据下载** | 无法导出 rankings、chunks、audit 等中间产物 |
| **无工作区浏览** | 无法直接查看 HTML、chunk、审计报告 |
| **无任务取消按钮** | 当前只能删除任务，不能优雅中断 |
| **导出格式有限** | 当前以 Markdown 为主 |

### 12.2 后端

#### 高优先级

| Gap | 说明 |
| :--- | :--- |
| **`tasks.json` 非事务性存储** | 并发场景存在损坏风险 |
| **无取消 API** | 无法显式中止长任务 |
| **无统一超时机制** | 长时间无输出任务缺少自动止损 |
| **无内置限速 / 重试策略** | 面对网络抖动与反爬时成功率不稳 |
| **可观测性弱** | 缺少结构化日志与关键 metrics |

### 12.3 Agent / 分析层

#### 高优先级

| Gap | 说明 |
| :--- | :--- |
| **无历史趋势分析** | 刷新更多是状态刷新，而不是时间序列洞察 |
| **无跨类目对比分析** | 当前仍是单类目视角 |
| **无竞争强度综合评分** | 用户无法一眼判断值不值得进入 |
| **无竞品追踪机制** | 不能指定 ASIN 持续跟踪变化 |

## 17. 结论

本项目已经具备从原型走向产品化的核心基础：Agent 流水线、Web 前端、SSE 进度推送、任务管理和报告读取能力均已存在。当前最重要的问题不是“有没有功能”，而是把 **路径管理、状态一致性、重跑策略、增量成本控制** 这几块地基打牢。

v1.0 的核心成功标准应定义为：

- **创建任务稳定可跑**
- **刷新逻辑明显省成本**
- **全量重新分析可用且不受旧状态污染**
- **前端能稳定看到正确 workspace 下的报告和下载内容**
- **删除 / 恢复 / 失败处理行为可预测**

在此基础上，再逐步扩展历史趋势、报告对比、多类目分析和协作能力。
