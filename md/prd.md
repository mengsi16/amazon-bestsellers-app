## 产品愿景与当前坐标

这是一个 **Amazon 细分类目 Bestsellers 自动化分析工具**：输入一个类目 URL，输出 5 个维度的结构化报告 + 汇总，帮助用户（很可能是亚马逊卖家或品牌方）快速摸清市场竞争格局。

核心价值主张：**零配置、一键启动、端到端自动 → 可操作的市场洞察报告**。

## 各层 Gap 分析

### 一、前端（用户侧体验）

#### 高优先级（直接影响留存）

| Gap                      | 说明                                                | 为什么重要                              |
| ------------------------ | --------------------------------------------------- | --------------------------------------- |
| **无任务搜索/过滤**      | Sidebar 只按时间倒序堆列表，100 个任务后无法定位    | 多类目用户核心痛点                      |
| **无报告对比**           | 不能同时打开两个类目的报告 Tab 对比                 | 卖家对比竞品类目的基础需求              |
| **无完成通知**           | 必须盯着浏览器等 30-90 分钟                         | 当前体验是"守株待兔"                    |
| **无 ETA / ASIN 级进度** | StageRail 只显示 5 个阶段，不知道还要跑多久、卡在哪 | 不确定性是用户流失最大原因              |
| **无成本展示**           | 报告 meta 有 `total_cost_usd`，UI 完全不显示        | Claude API 调用有成本，用户需要心里有数 |

#### 中优先级（体验完善）

| Gap                               | 说明                                                            |
| --------------------------------- | --------------------------------------------------------------- |
| **无原始数据下载**                | 只能下 .md 报告；rankings.jsonl、chunks 里的结构化数据无法导出  |
| **无工作区浏览**                  | 用户无法在 UI 里看 HTML、chunk、audit_report 等中间产物         |
| **无法中断任务**                  | 只能删除进程，不能优雅取消                                      |
| **无报告导出格式选择**            | 只有 .md；PDF/HTML/CSV 导出能覆盖更多使用场景                   |
| **ProgressPanel 未被使用**        | `ChatPanel.tsx` 和 `ProgressPanel.tsx` 在当前布局中处于废弃状态 |
| **前端轮询而非 SSE 推送任务列表** | 每 10s 轮询一次，切换任务时可能有延迟                           |

#### 低优先级（锦上添花）

| Gap                 | 说明                                 |
| ------------------- | ------------------------------------ |
| **无深色模式**      | 技术上简单，但用户长时间盯报告很重要 |
| **无报告书签/收藏** | 用户标记重要报告                     |
| **无移动端适配**    | Sidebar 折叠、报告排版自适应         |

### 二、后端（数据与 API 层）

#### 高优先级

| Gap                           | 说明                                   | 为什么重要                      |
| ----------------------------- | -------------------------------------- | ------------------------------- |
| **`tasks.json` 非事务性存储** | 并发写入有损坏风险；FastAPI 是多线程的 | 数据一致性是生产级底线          |
| **无任务取消 API**            | 前端无法优雅终止 subprocess            | 用户无法中途止损                |
| **无自动重试 + 退避**         | 爬虫偶发失败直接记录 error，不重试     | 弱网/亚马逊限流场景下成功率骤降 |
| **无内置限速**                | 爬虫无延迟控制，可能触发 Amazon 反爬   | 稳定性风险                      |
| **无 headless 浏览器复用**    | 每次 crawl 开新浏览器，启动成本高      | 大批量任务时速度极慢            |

#### 中优先级

| Gap                        | 说明                     |
| -------------------------- | ------------------------ |
| **CORS 全开（*）**         | 不适合带凭证的生产部署   |
| **FastAPI 未启用 OpenAPI** | 无交互式 API 文档        |
| **无数据库迁移机制**       | SQLite schema 变更靠手动 |
| **硬编码 `localhost`**     | 无法配置远端 backend URL |
| **无监控/可观测性**        | 无结构化日志、无 metrics |

#### 低优先级

| Gap                      | 说明                                                  |
| ------------------------ | ----------------------------------------------------- |
| **无结构化数据导出 API** | rankings.jsonl 等只能通过文件系统访问，API 层无法获取 |
| **无 Webhook 通知**      | 任务完成时推送到外部系统                              |
| **无多租户/用户隔离**    | 所有用户共享一个 tasks.json                           |

### 三、Agent 层（自动化流水线）

#### 高优先级

| Gap                      | 说明                                             | 为什么重要                                         |
| ------------------------ | ------------------------------------------------ | -------------------------------------------------- |
| **无多类目横向对比分析** | 每个类目单独跑，4 个 analyst 都是单类目视角      | **做竞争分析时跨类目对比是核心需求**，当前完全缺失 |
| **无历史趋势分析**       | 每次刷新只是"新 ASIN 来了"，没有时间序列数据     | 判断类目走势（季节性、竞争加剧）完全不可能         |
| **无价格/销量估算层**    | 报告基于页面文字，没有真实销量/GMV 估算模型      | 卖家最关心的"这个位置能出多少单"无法回答           |
| **无竞品 ASIN 追踪**     | 不能指定几个 ASIN 持续追踪变化                   | 监控竞品动态是高频需求                             |
| **无竞争强度综合评分**   | 只有各维度单独报告，没有统一的"进入难度"量化分数 | 卖家需要一句话判断：能进吗？                       |

#### 中优先级

| Gap                           | 说明                                                               |
| ----------------------------- | ------------------------------------------------------------------ |
| **无报告质量评分**            | audit 只审计 chunk 完整性，不审计报告本身                          |
| **chunker 自依赖 Playwright** | chunker 生成的代码运行在 subprocess 环境，依赖本地 Playwright 安装 |
| **无 ASIN 变体处理**          | 同 ASIN 的子变体（颜色/尺寸）是否被区分处理不清楚                  |
| **无类目层级自动识别**        | 用户必须手动找 Bestsellers URL，不能从任意 Amazon 页面推断         |
| **analyst 之间无信息共享**    | 4 个 analyst 完全独立运行，有大量重复读取同一份 chunk              |

#### 低优先级

| Gap                                    | 说明                       |
| -------------------------------------- | -------------------------- |
| **无自定义分析维度**                   | 只能跑预设的 5 个维度      |
| **无报告自动摘要到 Notion/飞书/Slack** | 报告出来后手动复制粘贴     |
| **无模型切换**                         | 只能用一个模型跑全程       |
| **无置信度标注**                       | 报告内容不标注信息来源强弱 |

## 做"完整 Amazon 细分类目分析"还缺什么

按用户旅程排列：

```
【用户输入】
  ↓
  ↓ 现状：只能输入一个 Bestsellers URL
  ↓ 缺：批量导入多个类目、统一分析
  ↓
【数据采集层】
  ↓
  ↓ 现状：爬 → chunk → extract
  ↓ 缺：
  ↓   · ASIN 历史排名数据（需要定期重新爬同一类目才能建时间序列）
  ↓   · 真实销量估算（基于排名区间换算，Amazon 未公开但可估算）
  ↓   · 评论详情（当前只爬了商品页，评论详情页没有系统抓取）
  ↓   · QA 数据（用户问答是强信号，当前未覆盖）
  ↓   · 类目结构（父子类目关系、节点树）
  ↓
【分析层】
  ↓
  ↓ 现状：单类目 × 4 个独立维度
  ↓ 缺：
  ↓   · 跨类目横向对比（我的产品能否同时进入 A 和 B 类目？）
  ↓   · 竞争强度综合评分（进入门槛 = f(头部集中度, 新品存活率, A+覆盖率, 价格区间密度)）
  ↓   · 机会缺口识别（哪些细分标签/长尾词没有人做 A+？）
  ↓   · 定价策略建议（基于价格分布 + 排名区间 → 推荐定价区间）
  ↓   · 季节性分析（需要历史数据才能做，当前时间序列缺失）
  ↓
【输出层】
  ↓
  ↓ 现状：5 个 Markdown 报告 + 1 个 summary
  ↓ 缺：
  ↓   · 可执行建议（不是描述现状，而是给出 action：定价 $XX, 主图加XX, 目标类目 YY）
  ↓   · 投资回报预估（结合竞争强度 + 定价 → 预估 ROI）
  ↓   · 一页纸 Executive Summary（给老板看的，不是给运营看的）
  ↓   · PDF 打包导出
  ↓   · 定时自动分析 + 报告推送
  ↓
【协作层】
  ↓
  ↓ 现状：单人单机器
  ↓ 缺：
  ↓   · 多用户 + 团队空间
  ↓   · 报告分享链接（只读）
  ↓   · 竞品追踪 + 变化提醒
  ↓   · 与 Shopify/卖家后台数据打通
```

## 优先级建议（如果只做最重要的 5 件事）

从 PM 角度，**按"用户价值 / 实现成本"排序**：

1. **多类目批量分析** — 用户最常见的工作流是"同时分析 5 个候选类目选最优"，这是当前最大的体验断点
2. **历史趋势数据 + 可视化** — 至少做 2 次刷新后的排名变化图，这是差异化核心
3. **竞争强度综合评分** — 把 4 个 analyst 的结论聚合成一个可操作的分数
4. **任务 ETA + ASIN 级进度** — 把"盯着屏幕 1 小时"变成"该干啥干啥，完成有通知"
5. **报告对比** — 两个类目并排看，是选类目的核心场景

后端的事务性存储和限速保护属于 **技术地基**，建议在任何规模化功能之前先修，否则多用户场景会出问题。
